Architecture Decision Guide

How to avoid API calls on every page load

Goal: Pull news once, store it, and serve reads from a local data store with predictable refresh windows.

Frontend: News.Traxxis.club Backend: NewsAPI.YubTub.club SQL Server Schema: NewsSite
Primary Win: Faster page loads + fewer API calls
Secondary Win: Better reliability during API outages
Tradeoff: Must define cache freshness rules

Option Comparison

A. Browser-only Cache

Store in localStorage or IndexedDB from frontend.

Complexity: Low
  • Quickest to build
  • Reduces repeat calls per browser
  • Not shared across users/devices

B. API Memory Cache

ASP.NET API keeps short-lived in-memory cache.

Complexity: Low-Med
  • Easy server-side win
  • One upstream fetch serves many users
  • Lost on app restart

C. SQL-backed Cache

Persist normalized news in SQL Server NewsSite schema.

Complexity: Medium
  • Durable across restarts
  • Auditable and queryable
  • Needs ingestion + expiration logic

D. Distributed Cache Layer

Add Redis on top of SQL/API for high scale.

Complexity: Med-High
  • Strong performance under load
  • Flexible TTL control
  • Extra infrastructure to manage

Recommended Path For Your Stack

Phase What to Implement Why It Helps Risk
1 API memory cache with 5-15 min TTL Immediate reduction in external API calls Low
2 Persist articles in SQL Server NewsSite schema Durable history and restart-safe data Medium
3 Optional browser cache for repeat visits Even faster UX + lower API traffic Low
4 Add Redis only if traffic/load requires it Scale read throughput without overcomplicating now Medium
Freshness Model

Store fetched timestamp and a computed expiresAt per article or query set.

Fallback Strategy

If upstream news API fails, serve last good dataset from SQL cache.

Data Governance

Use source+url hash dedupe keys and cleanup jobs for expired data.

SQL Schema Starter (Conceptual)

NewsSite.Source

  • SourceId (PK)
  • Name
  • BaseUrl

NewsSite.Article

  • ArticleId (PK)
  • SourceId (FK)
  • Title, Url, PublishedAt
  • Summary, Category, ImageUrl
  • ContentHash (dedupe)

NewsSite.CacheManifest

  • QueryKey
  • FetchedAt
  • ExpiresAt
  • Status / ErrorInfo

Use this as a planning model only. Final table design depends on query patterns, indexes, retention period, and legal/API usage terms.
