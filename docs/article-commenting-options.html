Next Branch Planning

Article Comments Without Full User Signup

You already persist articles in MySQL via news_articles. The next branch can add lightweight commenting by creating a pseudo-profile on first comment, seeded from randomuser.me, then gradually replacing parts of that profile when the visitor explicitly provides better data later.

1 Keep article comments scoped to stored articles only.
2 Preserve the randomuser payload shape you actually use.
3 Separate identity signals from display-profile data.

Working assumption

The first release supports top-level comments only. Replies, moderation queues, and account editing can land in later branches.

Design pressure

IP addresses and browser geolocation can help re-identify a returning visitor, but they are weak as primary identity keys and sensitive from a privacy standpoint.

Constraints From The Current Backend

The branch should fit the existing persisted-article model instead of rebuilding account infrastructure from scratch.

Already present

Stored article identity

Articles already have a durable article_hash. Comments should attach to the article row by foreign key, not by headline or URL text.

Needs care

Random user seed data

Only a subset is useful for a comment profile: display name, username, email placeholder, nat, login uuid, picture thumbnail, and a limited location object.

Main risk

Identity matching

Raw IP alone is unstable and often shared across multiple people. Geolocation alone is even weaker. Treat both as optional signals, not as the canonical account key.

Identity Strategy Options

The branch decision is less about comments themselves and more about how a future returning commenter gets mapped back to the same pseudo-account.

Recommended

A. Seeded profile + first-party visitor key

Create a pseudo-user when the first comment is submitted. Store a server-generated visitor key in an HttpOnly cookie and use it as the main return-path identifier.

  • Most stable for repeat comments
  • Least dependent on sensitive network data
  • Still allows optional IP/location enrichment
Possible

B. Hashed consented fingerprint

Hash normalized IP plus coarse geolocation plus an application salt. Reuse the same pseudo-user when the hash matches later.

  • Works without a cookie if signals stay stable
  • More privacy-sensitive
  • False matches and missed matches are likely
Not advised first

C. Raw IP/location as account identity

This is easy to explain but brittle in practice. NAT, VPNs, mobile network churn, and consent refusal will make the account experience inconsistent.

  • Higher privacy burden
  • Unreliable for long-term identity
  • Difficult to justify as the primary key

My Recommendation For The Next Branch

Build comments around a lightweight comment_users table plus a article_comments table, and keep consented identity evidence in separate columns or a separate table. Do not overwrite fake profile fields with IP address data just to make the profile re-identifiable.

The randomuser data should remain what it is: seed profile presentation. If a visitor later grants approximate location or eventually edits their own profile, that becomes first-party profile data, not a mutation of the original randomuser object. That approach keeps the model understandable and auditable.

If you want returning commenters before formal sign-up exists, use a server-issued visitor token as the primary lookup key. Then optionally store consented IP and geolocation snapshots for abuse analysis, soft matching, and later profile enrichment.

First branch scope

  • Create schema for pseudo-users and top-level comments
  • Support one profile per visitor token
  • Persist randomuser subset plus raw subset JSON
  • Store thumbnail bytes and MIME type only
  • Record consent flags for IP and geolocation
  • Skip replies, likes, moderation UI, and profile editing

Suggested Database Shape

Keep the schema intentionally small. Preserve only the randomuser structure you care about, and normalize what you need for queries.

Table Purpose Important columns
comment_users One pseudo-account per visitor or identity grouping. id, visitor_key_hash, source, display_name, username, email_placeholder, profile_thumbnail_blob, profile_thumbnail_mime, randomuser_login_uuid, nat, raw_profile_json, created_at, updated_at
comment_user_signals Consent-aware identity and location evidence, separate from display profile. id, comment_user_id, signal_type, signal_hash, ip_address_encrypted_or_null, country_code, region_name, city_name, latitude, longitude, consent_granted, captured_at
article_comments Top-level comments attached to persisted articles. id, article_id, comment_user_id, body, status, created_at, updated_at, deleted_at

If you want an even smaller first cut, comment_user_signals can be folded into comment_users, but separating it now makes future consent and audit handling much cleaner.

Randomuser Fields Worth Keeping

The payload does not need to be mirrored 1:1 in relational columns. Keep a subset in the same nested JSON structure, plus a few extracted columns for queries and display.

Keep Why Store as
name.title, name.first, name.last Display name construction Nested JSON + flattened display_name
login.uuid, login.username Stable upstream seed identifiers Nested JSON + indexed scalar columns
email Placeholder identity field until user edits profile Scalar column and JSON
location.country, state, city Light profile flavor only Nested JSON only unless you plan to query it
picture.thumbnail Immediate avatar use URL in JSON plus downloaded thumbnail bytes in BLOB
nat Harmless display metadata Scalar column and JSON

Thoughts On IP And Location

This is the part most likely to cause future cleanup if the first branch oversimplifies it.

Safe pattern

Use explicit consent flags

Persist whether IP capture and geolocation capture were granted. That matters if you later add export, deletion, or moderation workflows.

Better storage

Hash for matching, keep raw sparingly

If you store network-derived identity signals, use a salted hash for lookup. Keep raw IP only if you have a concrete operational reason and a retention plan.

Avoid

Do not map IP into profile name fields

Replacing fake display data with IP or coordinates inside the same profile object will make the model confusing and harder to reason about later.

Starter SQL Sketch

Not production-final, but this is a good branch target for the first migration set.

CREATE TABLE comment_users (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  visitor_key_hash CHAR(64) NOT NULL,
  source ENUM('randomuser_seed', 'user_enriched') NOT NULL DEFAULT 'randomuser_seed',
  display_name VARCHAR(191) NOT NULL,
  username VARCHAR(191) NULL,
  email_placeholder VARCHAR(191) NULL,
  randomuser_login_uuid CHAR(36) NULL,
  nat CHAR(8) NULL,
  profile_thumbnail_blob LONGBLOB NULL,
  profile_thumbnail_mime VARCHAR(64) NULL,
  raw_profile_json JSON NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uq_comment_users_visitor_key_hash (visitor_key_hash),
  KEY idx_comment_users_randomuser_login_uuid (randomuser_login_uuid)
);

CREATE TABLE comment_user_signals (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  comment_user_id BIGINT UNSIGNED NOT NULL,
  signal_type ENUM('ip', 'geolocation', 'combined') NOT NULL,
  signal_hash CHAR(64) NOT NULL,
  ip_address_value VARCHAR(64) NULL,
  country_code VARCHAR(8) NULL,
  region_name VARCHAR(191) NULL,
  city_name VARCHAR(191) NULL,
  latitude DECIMAL(9,6) NULL,
  longitude DECIMAL(9,6) NULL,
  consent_granted TINYINT(1) NOT NULL DEFAULT 0,
  captured_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_comment_user_signals_user_id (comment_user_id),
  KEY idx_comment_user_signals_signal_hash (signal_hash),
  CONSTRAINT fk_comment_user_signals_user
    FOREIGN KEY (comment_user_id) REFERENCES comment_users(id)
      ON DELETE CASCADE
);

CREATE TABLE article_comments (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  article_id INT UNSIGNED NOT NULL,
  comment_user_id BIGINT UNSIGNED NOT NULL,
  body TEXT NOT NULL,
  status ENUM('published', 'hidden', 'deleted') NOT NULL DEFAULT 'published',
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  deleted_at TIMESTAMP NULL,
  PRIMARY KEY (id),
  KEY idx_article_comments_article_id (article_id),
  KEY idx_article_comments_comment_user_id (comment_user_id),
  CONSTRAINT fk_article_comments_article
    FOREIGN KEY (article_id) REFERENCES news_articles(id)
      ON DELETE CASCADE,
  CONSTRAINT fk_article_comments_user
    FOREIGN KEY (comment_user_id) REFERENCES comment_users(id)
      ON DELETE RESTRICT
);

Suggested Branch Sequence

Keep the first branch narrow enough that it can be validated end-to-end without building a full social system.

Phase 1

Add SQL migrations for pseudo-users, comment signals, and article comments.

Phase 2

Add API endpoints to create a comment profile, post a comment, and list comments for one article.

Phase 3

Fetch one randomuser seed on first comment only, store the subset JSON, and download the thumbnail bytes.

Phase 4

Wire the frontend article detail or article card UI to render comments and submit a new one.
