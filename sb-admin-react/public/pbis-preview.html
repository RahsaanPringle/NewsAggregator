Repo API Setup (PAT vs JWT)

Use a PAT First (Recommended Now)

  • Fastest path for local development and prototyping.
  • No JWT signing implementation required.
  • Store token in server env only, never in browser code.
  • Great for read/list + basic edit actions while validating UX.

Use GitHub App + JWT Later

  • Best long-term for production and team-managed access.
  • Server signs JWT, exchanges for installation access token.
  • Better permission scoping, auditing, and token rotation.
  • More setup complexity but stronger operational model.
What to set up now (minimal):
Create a fine-grained PAT and grant repository + projects access for NewsAggregator. Add it to the backend env as GITHUB_TOKEN.
Suggested backend env keys:
GITHUB_TOKEN, GITHUB_OWNER, GITHUB_REPO, GITHUB_PROJECT_NUMBER.
Security rule:
Frontend should only call local routes like /api/pbis. The server route calls GitHub APIs and returns normalized card data.
Fine-Grained PAT Setup Checklist
  1. Open GitHub Settings -> Developer settings -> Personal access tokens -> Fine-grained tokens.
  2. Click Generate new token and set an expiration date.
  3. Under Resource owner, choose your account.
  4. Under Repository access, select Only select repositories and choose NewsAggregator.
  5. Grant repository permissions: Issues (Read and write) and Metadata (Read-only).
  6. Grant account/organization permission for Projects (Read and write) if your project board requires updates.
  7. Generate token and store it once in your backend env file as GITHUB_TOKEN.
  8. Restart backend and test a route like /api/pbis that reads from GitHub and returns normalized card data.
